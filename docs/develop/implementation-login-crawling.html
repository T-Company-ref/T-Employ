로그인 기반 수집 아키텍처

대상: 잡코리아 기업계정(인재검색/포지션 제안), 사람인 기업계정(인재풀), 공고 지원자 수집

문서 버전 v0.1 작성일 2026-07-07 목적: 운영 가능한 구현안 확정

1. 전제 및 제약

GitHub Pages는 정적 호스팅이므로 플랫폼 기업계정 로그인 주체가 될 수 없다. 로그인/크롤링은 서버 실행 주체(Worker/Runner)에서만 가능하다.
  • Pages 프런트는 수집 요청/상태조회 UI만 담당
  • 실제 로그인, 쿠키 보관, 크롤링은 GitHub Actions 또는 별도 서버에서 수행
  • 플랫폼의 2FA, 캡차, 비정상 로그인 탐지는 안정성 리스크

2. 권장 아키텍처 (실행안)

[GitHub Pages] 수집요청 버튼 / 상태조회 │ ▼ [GitHub API: workflow_dispatch] │ ▼ [Crawler Runner] Playwright 로그인 - 잡코리아 기업계정 - 사람인 기업계정 지원자/인재풀 데이터 수집 │ ▼ [DB + Object Storage] applicants, applications, talent_pool_candidates candidate_documents, crawl_jobs, crawl_logs │ ▼ [Pages] 수집 결과 시각화

2.1 실행 환경 선택

선택지장점단점권장도
GitHub Hosted Runner구성 간단, 초기비용 낮음IP 변동 큼, 로그인 차단 가능성 높음중
Self-hosted Runner (고정 IP)세션 안정성 높음, 차단 리스크 감소운영비/관리 필요상
별도 서버(Queue Worker)유연성 최고, 확장성 높음초기 구축비 큼상
권장: 1차는 Self-hosted Runner + GitHub Actions로 시작, 이후 트래픽 증가 시 Queue Worker로 확장.

3. 페이지 구성 기획 (T-Client 유사 + 차별화)

3.1 UI/브랜딩 방향

항목T-Client 공통 요소차별화 요소 (Employ)

컴포넌트필터 칩, 상태 배지, 우측 상세 패널추천 태그 타임라인, 면접 일정 캘린더, 노쇼/블락 상태 배지
색상 체계기존 중립 톤 + 브랜드 포인트채용 상태별 고정 컬러(합격/불합/노쇼/완료/제거)

3.2 페이지 정보구조

  1. 대시보드: 어제 수집건, 오늘 면접 일정, 추천 후보, 상태 변경 요약
  2. 신규 지원자: 공고 지원 기반 지원자 목록 + 즉시 추천 태그/면접 등록
  3. 인재검색 후보: 잡코리아 포지션 제안/사람인 인재풀 기반 후보 목록
  4. 면접 관리: 일정, 면접결과(합/불), 노쇼, 후속 액션 기록
  5. 운영 로그: 수집 성공률, 실패 내역, 메일 발송 내역

4. 로그인 문제 극복 전략

3.1 계정/인증 전략

  • 플랫폼별 전용 운영 계정 사용 (개인 계정 금지)
  • 2FA는 가능하면 TOTP 기반으로 통일, 백업코드 별도 보관
  • 비밀정보는 GitHub Secrets 또는 외부 Vault에서만 조회

3.2 세션 전략 (핵심)

전략구현효과
세션 재사용storageState.json 암호화 저장 후 재사용매번 로그인 회피, 차단 감소
세션 만료 감지수집 시작 시 로그인 페이지 리다이렉트 검사자동 재로그인 트리거
휴먼 개입 플로우캡차 발생 시 job 중단 + 운영자 승인 후 재개운영 지속성 확보
완전 무인 로그인은 플랫폼 정책/캡차 정책에 따라 실패 가능성이 높다. "자동 재시도 + 수동 복구"를 표준 운영 절차로 포함해야 한다.

5. GitHub Pages + GitHub Actions 연동 방식

4.1 Pages에서 가능한 것

  • workflow_dispatch 호출로 수집 작업 요청
  • crawl_jobs 조회로 상태 표시 (queued/running/succeeded/failed)
  • 수집 완료 데이터 렌더링

4.2 Pages에서 불가능한 것

  • 브라우저에서 기업계정 아이디/비밀번호를 직접 보관하고 로그인 실행
  • 보안 비밀(Secrets) 접근

4.3 구현 패턴

  1. Pages에서 "수집 시작" 클릭
  2. 서명된 요청으로 Actions 워크플로 트리거
  3. Runner가 Playwright 로그인/수집 수행
  4. DB 저장 후 결과 요약을 crawl_jobs.result_json에 기록
  5. Pages가 폴링하여 결과 반영

6. 워크플로 설계

5.1 Actions Workflow

워크플로트리거역할
crawl-applicants.yml매일 18:00 KST + manual공고 지원자 수집
crawl-talent-pool.yml매일 18:20 KST + manual잡코리아/사람인 인재풀 수집
compose-daily-report.yml매일 07:50 KST전일 데이터 집계/요약 본문 생성
send-daily-mail.yml매일 08:00 KST지정 수신자 메일 발송
refresh-session.ymlmanual + failure hook세션 재발급 전용

6.2 일일 배치 순서

  1. 18:00 공고 지원자 수집 시작 (사이트 순차 실행)
  2. 18:20 인재검색 후보 수집 시작 (잡코리아 → 사람인)
  3. 수집 종료 후 중복 정리/상태 동기화
  4. 07:50 전일 기준 보고 데이터 스냅샷 생성
  5. 08:00 지정 수신자에게 요약 메일 발송

6.3 Job 단계

  1. Lock 획득(concurrency + DB lock)으로 중복 실행 방지
  2. 비밀값 로드 (계정/암호/TOTP seed)
  3. 로그인 상태 검사 후 필요 시 재로그인
  4. 플랫폼별 목록/상세 수집
  5. PDF/이미지 스냅샷 저장 (Object Storage)
  6. DB upsert + 변경량 통계 생성
  7. 실패 시 재시도 및 운영자 알림

9. 보안 및 컴플라이언스

  • 기업 계정 자격증명은 Secrets/Vault에서만 관리
  • 민감 정보(이메일/전화번호/이력서)는 저장 시 암호화 또는 마스킹 정책 적용
  • 접근 로그 및 상태 변경 로그 1년 이상 보관
  • 플랫폼 이용약관 준수 범위 내 수집 정책 명문화

10. 단계별 구현 계획

Phase 1 (1~2주)

  • Actions 워크플로 골격 + Playwright 로그인 스크립트
  • DB 스키마(crawl_jobs, platform_sessions)
  • 직원 닉네임 기반 추천 태그 MVP

Phase 2 (2주)

  • 세션 재사용/만료 복구 로직
  • 지원자/인재풀 분리 수집 파이프라인 완성
  • Pages 수집요청/상태조회 연결 + 18:00 자동 배치

Phase 3 (1~2주)

  • 08:00 자동 리포트 및 실패 알림
  • 면접 일정/결과/노쇼/입사완료/블락 상태 운영
  • Self-hosted Runner 이중화

Phase 4 (선택)

  • Queue Worker 독립 배포
  • 플랫폼별 회복 시나리오 자동화 고도화
  • 추가 채용 플랫폼 확장

11. 최종 결론

  • GitHub Pages는 UI/요청만 담당, 로그인/수집은 Runner에서 수행
  • 운영 핵심 시간축은 18:00 순차 수집, 08:00 요약 발송
  • 추천 태그는 "누가 달았는지"를 staff_id(초기 닉네임)로 추적
  • 후보 제거는 삭제가 아닌 상태 기반 소프트 삭제로 관리
  • 핵심 난제는 로그인 안정성이며, 세션 재사용 + 수동 복구 절차가 필수
  • 실무 운영 관점에서 Self-hosted Runner 기반이 가장 현실적

12. 운영 체크리스트

12.1 계정/로그인 운영

  • 플랫폼별 운영 계정 만료일, 비밀번호 변경일, 2FA 백업코드 점검
  • 세션 만료율(최근 7일) 임계치 초과 시 자동 경고
  • 캡차/차단 발생 시 수동 복구 절차(runbook) 표준화

12.2 데이터 품질 운영

  • 중복 후보 병합 규칙: 이메일 > 전화번호 > 프로필 URL 순 우선
  • 상태 역전 방지 규칙: hired, blocked 상태는 승인자 없이 되돌림 금지
  • 문서(PDF) 수집 실패율, 프로필 파싱 실패율 일일 리포트 포함

12.3 알림/메일 운영

  • 메일 반송(bounce), 스팸 차단율 모니터링
  • 08:00 요약 메일 미발송 시 08:10 재시도, 이후 운영자 경고
  • 수신자 그룹 버전 관리(누가 언제 수신자 수정했는지 감사로그 기록)

12.4 보안/감사 운영

  • 추천 태그/상태 변경/블락 처리 행위는 전부 actor 기준 로그 보관
  • 민감정보 접근 로그 월간 검토
  • 개인정보 보관 만료 레코드 자동 파기 배치 실행

13. 멀티 사이트 확장 아키텍처

13.1 커넥터(Plugin) 표준 인터페이스

13.3 사이트 증가 시 운영 규칙

  • 사이트별 잡 격리: 한 사이트 실패가 전체 배치 중단시키지 않도록 분리 실행
  • 사이트별 제한값: rate_limit, timeout, retry_policy를 설정 테이블로 관리
  • 사이트 온보딩 체크리스트: 로그인 성공 > 목록 수집 > 상세 수집 > 정규화 > 헬스체크 통과

13.4 설정 기반 확장

테이블용도핵심 필드
platform_configs사이트 실행 설정platform, enabled, priority, crawl_window, rate_limit
platform_routes사이트 경로 버전 관리platform, route_name, selector_version, updated_at
platform_health사이트 상태 모니터링platform, last_ok_at, fail_count_24h, last_error
신규 사이트 추가 시 공통 파이프라인 변경 없이 connector 모듈 + platform_configs 등록만으로 온보딩 가능하도록 설계한다.
